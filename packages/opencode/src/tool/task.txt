Launch a subagent to handle a task synchronously (blocks until the subagent finishes).

Available agent types and the tools they have access to:
{agents}

You must specify a subagent_type parameter to select which agent type to use.

## CRITICAL: task vs background_task

You have TWO ways to launch subagents:

### `task` (this tool) — Synchronous, blocking
- Blocks until the subagent completes and returns the result.
- Use for quick, single operations where you need the result before your next step.
- **NEVER call multiple `task` tools in parallel.** This WILL crash. If you need parallel execution, use `background_task` instead.

### `background_task` — Asynchronous, non-blocking
- Returns immediately with a Task-N ID. The subagent runs in the background.
- Use when you have 2+ independent tasks to run concurrently.
- Use for long-running research, analysis, or multi-step operations.
- Check progress with `task_progress`, collect results with `wait_background_task`.

### Decision guide:
| Scenario | Use |
|----------|-----|
| Single quick lookup or operation | `task` |
| Need result before next step | `task` |
| 2+ independent tasks | `background_task` (for ALL of them) |
| Long-running research | `background_task` |
| Multi-file analysis | `background_task` |

## When to use task:
- When you need a single subagent result before continuing.
- When you are instructed to execute custom slash commands. Use the Task tool with the slash command invocation as the entire prompt. Example: Task(description="Check the file", prompt="/check-file path/to/file.py")

## When NOT to use task:
- **NEVER call multiple `task` tools in the same message.** Use `background_task` for parallel work.
- If you want to read a specific file path, use Read or Glob instead.
- If you are searching for a specific class definition like "class Foo", use Glob instead.
- If you are searching for code within a specific file or set of 2-3 files, use Read instead.

## Usage notes:
1. The result returned by the agent is not visible to the user. To show the user the result, send a text message back with a concise summary.
2. The output includes a task_id you can reuse later to continue the same subagent session.
3. Each invocation starts with a fresh context unless you provide task_id to resume a previous session.
4. Your prompt should contain a highly detailed task description. Specify exactly what information the agent should return.
5. Clearly tell the agent whether you expect it to write code or just to do research.
6. If the agent description mentions proactive use, use it without waiting for the user to ask.

## Example:

Launch an explore agent (single, blocking):
```
task({
  "description": "Find auth implementation",
  "prompt": "Search the codebase for all authentication-related files. Report file paths and a summary.",
  "subagent_type": "explore"
})
```

For parallel work, use background_task instead:
```
background_task({ "description": "Research API patterns", "subagent_type": "general-purpose" })
background_task({ "description": "Explore test coverage", "subagent_type": "explore" })
// Continue working, then later:
wait_background_task()  // collects all results
```
